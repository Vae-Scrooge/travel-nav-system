# 构建与部署体系

## 目标

建立可重复、可审计、可追溯的构建流程，减少手工编译带来的环境差异和发布风险。

## 标准构建流程

项目根目录提供 `Makefile`，推荐使用以下命令：

```bash
make build
make test
make dist
make clean
```

| 命令 | 作用 |
|---|---|
| `make build` | 编译主程序 |
| `make test` | 编译并执行自动化测试 |
| `make coverage` | 生成覆盖率数据 |
| `make dist` | 生成发布目录 |
| `make clean` | 清理构建产物 |

## 环境隔离

| 环境 | 用途 | 数据要求 |
|---|---|---|
| 开发环境 | 本地编码和调试 | 可使用小规模样例数据 |
| 测试环境 | 自动化测试和回归验证 | 使用固定测试数据 |
| 预发布环境 | 发布前验收 | 使用脱敏后的准生产数据 |
| 生产环境 | 正式运行 | 使用真实数据并启用备份 |

当前项目为本地控制台程序，暂未接入服务器部署。环境隔离主要通过不同数据目录、不同配置文件和不同构建产物实现。

## 构建产物管理

发布产物应包含：

1. 可执行文件。
2. 必要的数据文件模板。
3. 版本号或提交哈希。
4. CHANGELOG。
5. 运行说明。

建议发布目录结构：

```text
build/dist/
  toursys
  graphParams.txt
  graphVertex.txt
  graphEdge.txt
  user.txt
  CHANGELOG.md
```

## 版本追溯

每次发布应记录：

- Git 提交编号。
- 构建时间。
- 构建环境。
- 测试结果。
- 变更说明。

## 一键部署规划

当前阶段的一键部署目标是生成可运行目录：

```bash
make dist
```

后续可增加 `scripts/deploy.ps1` 或 `scripts/deploy.sh`，用于复制发布产物、备份旧版本、替换数据模板和输出部署日志。

## 审计要求

1. 所有发布必须来自版本控制中的代码。
2. 发布前必须通过 `make test`。
3. 构建产物不得手工修改。
4. 生产数据不得提交到代码仓库。
