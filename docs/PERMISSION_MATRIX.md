# 用户权限矩阵

## 角色定义

| 角色 | 说明 | 判定方式 |
|---|---|---|
| 游客 | 未登录或退出登录状态 | 未通过登录验证 |
| 普通用户 | 可查询和浏览景点信息 | 非 `admin` 用户登录成功 |
| 管理员 | 可维护景点和道路数据 | 用户文件第三字段为 `admin` |

说明：当前版本已经从“用户名规则”升级为“用户文件角色字段”规则。旧两字段用户记录会被兼容识别为普通用户。

## 权限矩阵

| 功能 | 游客 | 普通用户 | 管理员 |
|---|---:|---:|---:|
| 登录 / 注册 | 是 | 是 | 是 |
| 查看景点图 | 否 | 是 | 是 |
| 生成导游路线 | 否 | 是 | 是 |
| 检测导游路线环路 | 否 | 是 | 是 |
| 查询最短路径 | 否 | 是 | 是 |
| 查看访问统计 | 否 | 是 | 是 |
| 创建或替换景点图 | 否 | 否 | 是 |
| 景点新增、重命名、删除 | 否 | 否 | 是 |
| 道路新增、更新、删除 | 否 | 否 | 是 |

## 已实现控制点

1. 登录成功后通过 `userManagerSetCurrentUser` 记录当前用户和角色。
2. `userCanManageGraph` 统一判断当前用户是否具备图数据维护权限。
3. 主流程中创建或替换图数据、维护景点道路前均执行权限检查。
4. 景点和道路维护操作会写入 `audit/admin_actions.log`。
5. 帮助菜单可查看最近迁移审计和管理员操作审计记录。

## 审批与审计建议

1. 管理员权限变更应通过独立审批流程完成，不应由普通用户自助提升。
2. 用户文件中的角色字段建议纳入变更审计。
3. 后续如引入服务层，所有写操作接口必须复用同一套权限判断函数。
4. 权限校验失败应记录安全日志，便于排查越权尝试。

## 后续增强

1. 将角色字段进一步迁移到独立权限表。
2. 支持更多角色，例如运营员、审计员和只读管理员。
3. 为统计报表、用户管理和数据恢复增加更细粒度权限。
