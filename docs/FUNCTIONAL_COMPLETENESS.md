# 功能完整性梳理

## 目标

系统性梳理当前旅游景点管理系统的已实现功能、缺失功能和验收标准，确保后续完善工作有明确范围。

## 当前已实现功能

| 功能 | 状态 | 说明 |
|---|---|---|
| 用户注册 | 已实现 | 支持写入 `user.txt` |
| 用户登录 | 已实现 | 支持用户名和密码校验 |
| 图数据加载 | 已实现 | 从文本文件加载景点和道路 |
| 图数据保存 | 已实现 | 将景点和道路写回文本文件 |
| 景点网络展示 | 已实现 | 以邻接矩阵形式输出 |
| 导游路线生成 | 已实现 | 基于遍历结果生成路线 |
| 环路检测 | 已实现 | 对导游图进行检测 |
| 最短路径查询 | 已实现 | 单次查询使用 Dijkstra |
| 景点新增 | 已实现 | 通过 `graphAddSpot` 和管理菜单完成 |
| 景点删除 | 已实现 | 删除景点时同步删除相关道路 |
| 景点修改 | 已实现 | 通过 `graphRenameSpot` 修改景点名称 |
| 道路新增、删除、修改 | 已实现 | 通过 `graphAddOrUpdateRoad` 和 `graphDeleteRoad` 完成 |
| 访问统计 | 已实现 | 记录系统启动、登录、注册、路径查询和景点访问次数 |
| 密码哈希存储 | 已实现 | 新用户使用 PBKDF2-HMAC-SHA256 存储 |

## 待补齐功能

| 功能 | 优先级 | 验收标准 |
|---|---|---|
| 管理员角色 | 中 | 管理员可维护景点图，普通用户只能查询 |
| 数据备份恢复 | 中 | 修改数据前自动备份，支持恢复 |
| 帮助菜单 | 低 | 用户可查看功能说明和输入格式 |
| RESTful API 服务层 | 中 | 引入 HTTP 服务后对外提供标准 RESTful 接口 |

## 边缘场景

必须覆盖以下异常场景：

1. 图数据为空。
2. 图数据文件缺失。
3. 图数据文件格式错误。
4. 景点名称重复。
5. 道路长度为 0 或负数。
6. 起点或终点不存在。
7. 起点和终点相同。
8. 两个景点之间不可达。
9. 用户重复注册。
10. 用户数据文件不可写。

## 用户验收测试

用户验收测试应至少验证：

1. 新用户可以注册并登录。
2. 管理员可以维护景点和道路。
3. 普通用户可以查询最短路径。
4. 系统对错误输入给出明确提示。
5. 修改后的图数据可以保存并重新加载。
6. 查询结果与人工计算结果一致。

## 完成标准

功能完善工作完成时，应满足：

- 所有高优先级功能实现并通过测试。
- 所有核心异常场景有明确处理。
- README 和 CHANGELOG 同步更新。
- 自动化测试覆盖率达到 80% 以上。
- 用户验收测试全部通过。
