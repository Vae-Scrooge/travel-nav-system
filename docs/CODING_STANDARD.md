# 代码规范

## 目标

统一项目代码风格，提高可读性、可维护性和可审查性，减少技术债务。

## 命名规范

| 类型 | 规则 | 示例 |
|---|---|---|
| 函数 | 使用小驼峰或动宾结构，保持现有风格一致 | `initGraph`、`freeGraph` |
| 变量 | 使用清晰英文名，避免无意义缩写 | `nodeCount`、`edgeLength` |
| 常量 | 使用全大写加下划线 | `MAXNUM`、`INF` |
| 结构体 | 使用名词，表达业务含义 | `ALGraph`、`CNode` |
| 文件 | 按模块命名 | `graph.c`、`userManager.c` |

## 格式规范

1. 使用 4 个空格缩进。
2. 每个函数只做一类事情。
3. 单个函数建议不超过 80 行。
4. 条件判断必须使用大括号。
5. 指针声明风格保持一致，例如 `ALGraph *graph`。
6. 头文件必须使用 include guard。

## 注释规范

推荐保留以下注释：

- 复杂算法的关键步骤。
- 非显而易见的边界处理。
- 文件格式说明。
- 外部接口的返回值约定。

应清理以下注释：

- 大段废弃代码。
- 与当前逻辑不一致的历史注释。
- 无实际信息量的注释。
- 乱码或编码异常的注释。

## 错误处理规范

1. 文件打开失败必须返回明确错误。
2. 内存分配失败必须有保护逻辑。
3. 数组下标使用前必须校验范围。
4. 字符串输入必须限制长度。
5. 用户输入错误时应提示并允许重试。

## 模块设计原则

| 模块 | 设计要求 |
|---|---|
| `global` | 只放公共类型、常量和通用工具函数 |
| `graph` | 只负责图结构创建、释放、加载、保存和转换 |
| `travels` | 只负责遍历、路径规划和环路检测 |
| `userManager` | 只负责用户注册、登录和校验 |
| `main` | 只负责流程组织，不承载复杂业务逻辑 |

## 自动化检查

本项目是 C 语言项目，不适用 ESLint 和 Prettier。建议使用以下工具：

| 工具 | 作用 |
|---|---|
| `gcc -Wall -Wextra -Wpedantic` | 编译期警告检查 |
| `clang-format` | 自动格式化 C 代码 |
| `cppcheck` | 静态分析 |
| `gcov` | 覆盖率统计 |

## 代码审查清单

每次合并前应检查：

1. 是否通过编译。
2. 是否通过自动化测试。
3. 是否新增或更新相关测试。
4. 是否影响数据文件格式。
5. 是否引入安全风险。
6. 是否更新 README 或 CHANGELOG。

## 技术债务管理

建议建立技术债务清单，至少记录：

- 问题描述。
- 影响范围。
- 优先级。
- 负责人。
- 计划完成时间。
- 当前状态。
