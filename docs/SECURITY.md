# 安全存储与数据保护方案

## 当前安全现状

当前系统使用 `user.txt` 保存用户信息。历史数据中可能存在“用户名 明文密码”格式；本轮改造后，新注册用户将保存为 PBKDF2-HMAC-SHA256 哈希格式，不再保存明文密码。

## 安全目标

1. 用户密码不得明文存储。
2. 配置和密钥不得硬编码在源码中。
3. 数据文件需要具备备份和恢复机制。
4. 如系统未来接入网络通信，应使用传输层安全协议（TLS/SSL）。
5. 敏感数据的访问和修改需要可追踪。

## 密码存储策略

当前已落地方案：

- 使用 PBKDF2-HMAC-SHA256。
- 每个新用户生成独立盐值。
- 密码校验时只比较哈希结果，不保存新用户原始密码。
- 为兼容历史数据，登录验证仍支持旧明文记录。
- 旧明文用户在登录验证成功后自动迁移为 PBKDF2-HMAC-SHA256 哈希格式。
- 盐值生成优先使用系统级安全随机源，避免继续依赖 `rand()` 生成盐值。
- 旧密码迁移前会在 `backup/user/` 下创建时间戳备份文件。
- 迁移过程会写入 `backup/user/migration.log`，记录用户文件、用户名、状态、影响记录数和耗时。
- 如果替换原用户文件失败，系统会尝试使用备份文件回滚，降低迁移中断导致用户数据损坏的风险。
- 管理员维护景点和道路时会写入 `audit/admin_actions.log`，记录操作者、角色、动作、目标和结果。
- 帮助菜单提供最近迁移审计和管理员操作审计记录查询入口。

推荐生产增强方案：

- 优先接入经过审计的 bcrypt、Argon2 或 PBKDF2 第三方库。
- 为权限变更增加独立审批和更细粒度审计。
- 对随机盐生成增加重复率检测和异常告警。

建议的用户文件格式：

```text
username algorithm$iterations$salt$hash
```

示例：

```text
admin pbkdf2_sha256$10000$random_salt$password_hash
```

说明：当前项目未引入第三方加密库，因此本轮采用项目内 PBKDF2-HMAC-SHA256 实现。若要达到更高安全等级，建议接入经过审计的密码哈希库并完成旧数据迁移。

## 配置和密钥管理

如果后续增加 API 密钥或数据库连接信息，应优先使用：

1. 环境变量。
2. 本地 `.env` 文件，并确保 `.env` 不提交到仓库。
3. 操作系统密钥管理服务。
4. 云厂商安全密钥管理服务。

## 数据备份策略

| 数据 | 备份频率 | 保留策略 |
|---|---|---|
| 图数据文件 | 每次修改前自动备份 | 至少保留最近 10 份 |
| 用户数据文件 | 每次注册或修改前备份 | 至少保留最近 10 份 |
| 发布产物 | 每次发布归档 | 保留所有正式版本 |

推荐备份目录：

```text
backup/
  graph/
  user/
  release/
```

## 恢复机制

恢复流程应包含：

1. 停止当前程序。
2. 校验备份文件是否完整。
3. 替换损坏的数据文件。
4. 启动程序并执行基础功能验证。
5. 记录恢复时间、恢复人和恢复原因。

当前旧密码迁移流程已具备自动备份和失败回滚的基础能力。人工恢复时可优先查看：

```text
backup/user/
backup/user/migration.log
```

## 传输安全

当前项目为本地控制台程序，不涉及网络传输。如果未来增加 Web 服务、远程登录或接口访问，必须启用 TLS/SSL，禁止明文传输用户名、密码和令牌。

## 合规建议

1. 最小化保存用户敏感信息。
2. 不在仓库中提交真实用户数据。
3. 明确数据备份和删除策略。
4. 对用户数据操作保留审计日志。
