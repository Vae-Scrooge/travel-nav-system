### 2.2 功能模块间的协调性（8.5/10）

#### 评价依据

项目采用清晰的模块化目录结构：

```
src/
  core/     # 核心数据结构与算法（全局、图、路径、输入、密码）
  console/  # 用户交互层（菜单、图管理控制台、路径查询控制台、用户控制台）
  manager/  # 业务逻辑层（用户管理、统计、备份）
```

**量化指标：**
- 模块数：3 个功能模块 + 1 个入口文件（main.c）
- 头文件保护：100%（所有 8 个 .h 文件均使用 `#ifndef/#define/#endif`）
- 循环依赖：0（依赖方向：main.c → console/ → manager/ → core/）
- 跨模块 extern 引用：2 处（`parray`、`guidePath`）

**质量评估：**
- **依赖方向合理**：`main.c` → `console/` → `manager/` → `core/`，无循环依赖。
- **接口封装规范**：各模块通过头文件暴露公共 API，内部函数使用 `static` 限定。
- **头文件保护完善**：所有 `.h` 文件均使用 `#ifndef/#define/#endif` 防止重复包含。
- **全局状态集中管理**：当前用户、全局统计等状态封装在各自模块的静态变量中，通过访问器函数暴露。

#### 发现的问题

1. **全局变量 `parray` 与 `guidePath` 的跨模块耦合**：`parray`（邻接矩阵）在 `graph.c` 中定义，被 `graph_console.c`、`travels.c`、`travels_console.c` 多处通过 `extern` 引用；`guidePath` 在 `graph.c` 中定义，被 `travels.c` 使用。这种隐式共享降低了模块独立性。
2. **`userManagerRoleForUsername` 为桩函数**：始终返回 `USER_ROLE_USER`，未实现基于运行时的角色查询（虽然文件版本已实现）。

#### 改进建议

- [P2] 将 `parray` 封装为图对象的成员，或提供显式的 `getMatrix`/`setMatrix` 访问接口，移除跨模块 `extern`。
- [P3] 统一角色查询入口， deprecate `userManagerRoleForUsername` 或使其代理到文件版本。

### 2.3 用户交互流程的合理性（7.0/10）

#### 评价依据

- 输入函数安全性：`readInt` 使用 `strtol`，`readString` 限制长度 255

**质量评估：**
- **菜单层次清晰**：主菜单 12 项功能（0-11）分组合理，权限敏感操作（1、7）前置检查。
- **危险操作双重确认**：重建景点图、删除景点、删除道路等操作均要求输入 "yes" 确认。
- **错误提示明确**：区分 "ERROR"（操作失败）、"Permission denied"（权限不足）、"Unknown scenic spot"（数据不存在）。
- **帮助系统完善**：选项 11 提供功能说明与审计日志快捷查看。

#### 发现的问题

1. **输入读取不支持空格**：`readString` 基于 `scanf("%255s")`，无法读取含空格的景点名称或描述（虽然描述读取在 `graph_console.c` 中通过单独调用实现，但本质上仍受限于空白分隔）。
2. **密码输入明文显示**：`readString` 读取密码时无回显隐藏，存在安全风险。
3. **异常输入恢复不够优雅**：当 `readInt` 遇到非法输入时，会返回 0 并提示，但 `scanf` 残留字符可能导致后续读取行为异常（已通过 `readString` 封装缓解，但未完全解决）。
4. **无数据变更未保存提示**：用户在执行修改后若直接退出（Ctrl+C），不会触发 `statsSave` 或 `saveGraph`。

#### 改进建议

- [P1] 为密码输入实现无回显读取（POSIX `termios` / Windows `SetConsoleMode`）。
- [P2] 使用 `fgets` 替代 `scanf` 族函数作为底层输入方式，支持空格并更好地控制缓冲区。
- [P2] 注册流程结束后自动提示"是否立即登录"，减少用户操作步骤。
- [P3] 注册信号处理函数（`atexit` / `signal`），在异常退出时尝试保存统计数据。

### 2.5 异常处理机制的完善程度（7.5/10）

#### 评价依据

**量化指标：**
- NULL 参数检查覆盖率：100%（所有公共 API 均对关键参数进行 NULL 检查）
- 文件操作回滚：2 处（`graphLoadFromFiles`、`userChangePasswordInFile`）
- 内存分配检查：3 处（`addEdge`、`transToMatrix`、`hashInsert`）
- 文件格式校验：4 项（节点数越界、重复名称、边数据多余字段、负数长度）

**已完善的方面：**

- **空指针防御**：所有公共 API（`graphAddSpot`、`graphDeleteRoad`、`dijkstraShortestPath` 等）均对关键参数进行 NULL 检查。
- **文件操作回滚**：`graphLoadFromFiles` 在解析失败时调用 `initGraph` 回滚；`userChangePasswordInFile` 与 `migrateLegacyUserPassword` 在替换失败时从备份恢复。
- **输入校验**：`readInt` 使用 `strtol` 并检测非法后缀；`readString` 限制长度防止溢出。
- **内存分配检查**：`malloc` / `calloc` 返回值在关键路径中被检查（如 `addEdge`、`transToMatrix`）。
- **文件格式校验**：`graphLoadFromFiles` 检测节点数越界、重复名称、边数据多余字段、负数长度等。

**存在的不足：**

1. **哈希表插入不检查重复键**：`hashInsert` 直接在链表头部插入，若因逻辑错误导致重复插入同一键，会使 `hashFind` 返回最旧值（虽然实际业务层通过 `locate` 前置检查避免了此问题）。
2. **部分内存分配失败处理不完整**：`hashInsert` 中 `malloc` 失败时静默返回，调用方无法感知哈希插入失败。
3. **transToMatrix 部分分配失败时的泄漏**：当 `parray[i]` 分配失败调用 `freeMatrix` 时，`parray` 本身及已分配的 `parray[0..i-1]` 会被释放，但调用方（如 `printGraph`）未收到错误码，可能继续使用 NULL 的 `parray`。
4. **scanf 残留问题**：虽然 `readString` 封装了一层，但当用户输入超长数据（超过 255 字节）时，`scanf` 仍可能截断并遗留未消费字符。
5. **无信号/异常退出处理**：未注册 `SIGINT` 处理函数，用户按 Ctrl+C 会导致内存泄漏与数据丢失。

#### 改进建议

- [P2] 为 `hashInsert` 增加返回值（`int`），使调用方可检测插入失败；或增加重复键断言。
- [P2] 统一 `transToMatrix` 的错误传播机制，使其返回 `int` 状态码，调用方根据返回值决定是否继续。
- [P3] 使用 `fgets` + `sscanf` 替代直接 `scanf`，彻底消除输入残留问题。
- [P3] 注册 `atexit` 处理器，在异常退出时保存统计数据并释放全局矩阵内存。

### 2.6 性能表现及资源占用情况（8.5/10）

#### 评价依据

**已实施的优化：**

| 优化项 | 优化前 | 优化后 | 提升效果 |
|---|---|---|---|
| 节点查找 | O(n) 线性扫描 | O(1) 哈希表（djb2 + 链地址法） | 大规模图显著加速 |
| Dijkstra 最短路径 | O(V^2) 数组扫描 | O((V+E) log V) 二叉最小堆 | 稠密图/大规模图查询速度提升 |
| 内存分配（islooptest） | 动态堆分配 | 栈上固定数组 | 减少堆碎片与分配开销 |

**性能基准测试结果（来自 PERFORMANCE_REPORT.md）：**

| 场景 | 优化前耗时 | 优化后耗时 | 提升幅度 |
|---|---|---|---|
| chain-large (100节点) | ~14ms | ~6ms | ~57% |
| dense (60节点全连接) | ~14ms | ~8ms | ~43% |
| unreachable (80节点) | ~4ms | ~1ms | ~75% |

**资源占用评估：**

- **内存结构紧凑**：邻接表存储，每条边仅 3 个字段（index、length、next 指针）。
- **哈希表开销可控**：`HASH_SIZE = 127`，每个条目约 20 字节，100 节点图的哈希开销约 2.5KB。
- **邻接矩阵按需分配**：仅在需要时通过 `transToMatrix` 构建，退出时释放。

#### 发现的问题

1. **transToMatrix 频繁重建**：每次查询最短路径、打印图、生成导游路线前均调用 `transToMatrix`，若图未修改则属重复计算。当前实现每次都会 `freeMatrix` + 重新 `calloc`/`malloc`。
2. **Floyd-Warshall 仍为 O(V^3)**：对于 100 节点的图，每次查询全源最短路径需 1,000,000 次松弛操作。虽然项目需求要求此算法，但在频繁查询场景下可考虑缓存结果。
3. **栈数组大小固定为 MAXNUM (100)**：虽然避免了动态分配，但 `MAXNUM` 为编译期常量，无法灵活扩展。

#### 改进建议

- [P2] 引入"脏标记"机制：当图结构未改变时，跳过 `transToMatrix` 的重复重建。
- [P2] 在 `shortPath` 结果上增加缓存，仅在图数据修改后失效，减少重复 Floyd-Warshall 计算。
- [P3] 考虑使用动态扩容的邻接矩阵，以支持运行时调整最大节点数（需权衡简单性与灵活性）。
