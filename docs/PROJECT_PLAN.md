# 系统性完善项目计划

## 总体目标

围绕自动化测试体系、构建部署体系、安全存储机制、功能完整性和代码规范化五个关键领域，对旅游景点管理系统进行阶段性完善，形成可测试、可构建、可维护、可扩展的工程化项目。

## 阶段计划

### 第一阶段：工程基础设施

周期：第 1 周

交付物：

- `Makefile`
- `tests/test_runner.c`
- `docs/TESTING.md`
- `docs/BUILD_DEPLOY.md`
- 持续集成配置

成功标准：

- 可以通过统一命令编译项目。
- 可以通过统一命令执行测试。
- 核心图结构和路径算法具备基础单元测试。

### 第二阶段：功能补齐

周期：第 2 至第 3 周

交付物：

- 景点新增、删除、修改功能。
- 道路新增、删除、修改功能。
- 用户访问统计功能。
- 管理员和普通用户角色区分。
- 用户验收测试清单。

成功标准：

- 高优先级功能全部实现。
- 核心异常场景全部有处理。
- 用户验收测试通过。

### 第三阶段：安全改造

周期：第 4 周

交付物：

- 密码哈希存储方案。
- 用户数据迁移脚本或迁移说明。
- 数据备份和恢复机制。
- 安全测试用例。

成功标准：

- 不再明文保存新用户密码。
- 旧用户数据有明确迁移策略。
- 数据文件修改前可自动备份。

### 第四阶段：质量提升

周期：第 5 周

交付物：

- 代码格式化规则。
- 静态检查配置。
- 覆盖率报告。
- 技术债务清单。

成功标准：

- 自动化测试覆盖率达到 80% 以上。
- 编译警告数量可控并持续下降。
- 关键模块复杂度降低。

### 第五阶段：发布验收

周期：第 6 周

交付物：

- 发布包。
- 版本记录。
- 用户手册。
- 验收报告。

成功标准：

- 构建产物可追溯。
- 发布流程可重复。
- README、CHANGELOG 和测试文档同步更新。

## 进度跟踪机制

建议使用看板管理以下状态：

| 状态 | 含义 |
|---|---|
| 待办 | 已识别但尚未开始 |
| 进行中 | 正在实现或验证 |
| 阻塞 | 依赖未满足或存在风险 |
| 待验收 | 已完成开发，等待验证 |
| 已完成 | 通过验收并归档 |

## 质量门禁

每个阶段完成前必须满足：

1. 代码可以编译。
2. 自动化测试通过。
3. 文档同步更新。
4. 新增功能有测试或验收记录。
5. 安全风险有明确说明。

## 风险与应对

| 风险 | 影响 | 应对措施 |
|---|---|---|
| 本地缺少 C 编译器 | 无法执行编译和测试 | 安装 GCC、Clang 或 MSVC，并通过 CI 补充验证 |
| 旧数据文件格式不稳定 | 加载失败或结果错误 | 增加格式校验和备份机制 |
| 密码迁移影响登录 | 用户无法登录 | 提供兼容校验和一次性迁移策略 |
| 功能扩展导致模块耦合 | 维护成本上升 | 保持图、用户、路径、主流程职责分离 |
| 缺少测试数据 | 覆盖率不足 | 建立 `tests/fixtures` 测试数据目录 |
