# 自动化测试策略

## 目标

建立覆盖核心业务逻辑的自动化测试体系，降低图数据管理、路径规划、用户管理和文件持久化修改后的回归风险。阶段性测试覆盖率目标为 80% 以上。

## 测试分层

| 类型 | 范围 | 当前落地方式 | 后续完善 |
|---|---|---|---|
| 单元测试 | `locate`、`addEdge`、`getlength`、`shortPath` 等纯逻辑函数 | `tests/test_runner.c` | 增加用户校验、文件解析、异常路径测试 |
| 集成测试 | 图加载、图保存、路径查询、用户注册登录组合流程 | 暂以测试用例清单管理 | 增加独立测试数据目录和脚本 |
| 端到端测试 | 控制台菜单完整交互流程 | 暂未自动化 | 使用输入重定向模拟菜单操作 |
| 性能测试 | 不同规模景点图下的路径查询耗时 | 暂未自动化 | 增加 10、50、100 个景点的测试数据 |

## 覆盖率目标

- 第一阶段：核心算法和图操作覆盖率不低于 60%。
- 第二阶段：核心模块整体覆盖率不低于 80%。
- 第三阶段：新增功能必须同步新增测试，避免覆盖率下降。

## 执行方式

安装 GCC 后执行：

```bash
make test
```

如需生成覆盖率信息：

```bash
make coverage
```

## 测试用例管理

测试用例应记录以下信息：

| 字段 | 说明 |
|---|---|
| 用例编号 | 例如 `GRAPH-001` |
| 测试目标 | 被验证的功能点 |
| 前置条件 | 需要准备的数据或状态 |
| 输入数据 | 测试输入 |
| 预期结果 | 成功标准 |
| 实际结果 | 执行结果 |
| 状态 | 通过、失败、阻塞 |

## 必测场景

1. 景点定位成功和失败。
2. 合法道路和非法道路添加。
3. 邻接矩阵转换。
4. 两点间最短路径计算。
5. 不存在景点输入。
6. 不可达路径。
7. 用户注册、重复注册和登录失败。
8. 图数据文件缺失或格式错误。

## 持续集成要求

每次提交或合并前应自动执行：

1. 编译检查。
2. 单元测试。
3. 覆盖率统计。
4. 基础静态检查。

当前项目已提供 `.github/workflows/ci.yml` 作为持续集成入口。
