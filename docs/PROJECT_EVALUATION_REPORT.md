# 项目评分评估报告

## 评估对象与范围

本次评估对象为 `toursys` 旅游景点管理系统当前工作区版本，范围包括：

1. 核心源码：`graph.c`、`travels.c`、`userManager.c`、`password.c`、`stats.c`、`global.c` 等。
2. 工程体系：`Makefile`、`.github/workflows/ci.yml`、`.gcovr.cfg`。
3. 测试与质量报告：`tests/test_runner.c`、`docs/TEST_REPORT.md`、`docs/STATIC_ANALYSIS_REPORT.md`。
4. 功能与设计文档：`README.md`、`docs/FUNCTIONAL_COMPLETENESS.md`、`docs/API_DESIGN.md`、`docs/SECURITY.md`、`CHANGELOG.md`。

本报告采用百分制，总分 100 分。评分遵循可验证证据优先原则：优先依据编译、测试、覆盖率、静态分析、文档和源码检查结果，避免仅凭主观印象评分。

本次为优化实施后的复评版本，重点核查上一轮改进项：文档一致性、CI 覆盖率硬门禁、编码治理、用户迁移安全、权限边界、服务层架构决策和性能基准体系。

## 主要优势

1. 工程质量显著提升：已经具备 Makefile、CI、覆盖率硬门禁、覆盖率统计和静态分析报告。
2. 核心模块测试覆盖充分：`graph.c`、`travels.c`、`userManager.c` 均超过 90% 行覆盖率。
3. 关键安全风险已下降：新用户不再保存明文密码，旧明文密码支持带备份、日志和回滚的迁移。
4. 路径查询性能更合理：单次最短路径查询使用 Dijkstra，避免不必要的全图最短路径计算。
5. 模块边界更清晰：控制台交互层与业务逻辑逐步分离，后续维护、测试和服务层抽象成本降低。
6. 权限边界进一步规范：管理员角色来自用户文件角色字段，管理员操作会写入审计日志。

## 主要不足

1. 当前乱码扫描未发现替换字符或典型乱码模式，但历史数据文件和外部导入数据仍需按 UTF-8 无 BOM 规范持续检查。
2. P1 模块 90% 覆盖率门禁尚未在 CI 中逐项硬化。
3. 密码迁移和管理员操作已有日志，但缺少权限变更审批和更完整审计检索界面。
4. 权限系统已有角色字段，但尚未独立成权限表。
5. RESTful API 目前仅有设计文档和架构决策，尚未实现真实 HTTP 服务层。
6. 备份恢复入口和交互式端到端自动化测试仍属于待补齐内容。

## 改进建议

1. 将 UTF-8 无 BOM 检查纳入后续提交前巡检，防止历史数据或外部导入内容再次引入乱码。
2. 在 CI 中逐步增加 P1 模块 90% 分模块门禁。
3. 将用户角色从用户文件角色字段进一步升级为独立权限表。
4. 增加权限变更审批和更完整的审计检索界面。
5. 若确有多端访问需求，再基于现有业务函数实现独立服务层，而不是在菜单逻辑上直接拼接接口。
6. 增加更统一的错误恢复指引和操作结果提示。
7. 扩展性能基准，记录随机图、密集图、不可达图、内存占用和 CPU 使用率。

## 评估结论

当前项目已从早期课程型控制台程序，提升为具备基础工程化、自动化测试、安全密码存储和静态分析流程的较完整 C 项目。以当前证据看，项目核心功能稳定性和可维护性已经达到较好水平，可以支撑继续迭代。

综合判断：当前版本评分为 **91.2/100，折算 9.1/10**。下一阶段最值得优先处理的是 P1 分项门禁、权限表升级、权限变更审批、备份恢复入口和服务层需求确认。
