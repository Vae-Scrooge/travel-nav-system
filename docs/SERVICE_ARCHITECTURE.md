# 服务层架构决策说明

## 当前决策

当前项目仍是本地 C 控制台程序，尚未确认存在明确多端访问需求。因此本轮不启动 HTTP 服务层建设，避免为了接口形式过早引入网络框架、路由、序列化和部署复杂度。

## 启动服务层的前置条件

只有满足以下任一条件时，才建议进入服务层建设阶段：

1. 需要 Web、移动端或其他客户端远程访问同一套景点数据。
2. 需要多人并发使用系统。
3. 需要把当前控制台程序改造为前后端分离架构。
4. 需要与第三方系统进行接口集成。

## 分层原则

若后续确认需要服务层，应采用以下分层：

| 层级 | 职责 |
|---|---|
| 控制台层 | 负责菜单、输入输出和用户提示 |
| 服务层 | 负责接口契约、参数校验、权限校验和统一错误码 |
| 业务层 | 复用现有图 CRUD、路径查询、用户管理和统计函数 |
| 数据层 | 负责文本文件或未来数据库的读写 |

服务层不得直接拼接在菜单逻辑中；控制台层和服务层都应调用同一批业务函数。

## 候选接口契约

| 接口 | 业务函数 |
|---|---|
| `GET /spots` | `graphLoadFromFiles` / 图遍历输出 |
| `POST /spots` | `graphAddSpot` |
| `PUT /spots/{name}` | `graphRenameSpot` |
| `DELETE /spots/{name}` | `graphDeleteSpot` |
| `POST /roads` | `graphAddOrUpdateRoad` |
| `DELETE /roads/{from}/{to}` | `graphDeleteRoad` |
| `GET /paths/shortest` | `dijkstraShortestPath` |
| `GET /stats` | `statsGetSnapshot` |

## 错误码建议

| 错误码 | 含义 |
|---|---|
| `0` | 成功 |
| `1001` | 参数缺失或格式错误 |
| `1002` | 资源不存在 |
| `1003` | 资源冲突 |
| `1004` | 权限不足 |
| `1005` | 数据持久化失败 |
| `2001` | 路径不可达 |

## 当前落地项

1. 已将控制台交互从核心业务中拆出，降低未来服务层复用成本。
2. 已建立权限判断函数，未来服务接口可直接复用。
3. 已有 `docs/API_DESIGN.md` 作为 RESTful 映射草案。
